Share & Swap Shop

2026 RMIT Programming Project Real client work Most recent

A full-stack website for a Melbourne pop-up swap collective. Events sync automatically from the client's existing Humanitix ticketing account — zero manual website updates required, ever.

The problem

A non-technical community team needed a polished website where event listings stayed accurate without them touching code. They already used Humanitix for ticketing — the site had to be the single source of truth without becoming a maintenance burden.

The approach

Three phases before any production code: a platform comparison report (WordPress vs Squarespace vs Wix Studio with Velo), then complete UX wireframes for all 8 pages with a Figma design token system, then the build.

What I did, specifically

  1. API integration Server-side .jsw backend authenticating to Humanitix's REST API and normalising messy responses (locations as strings or objects, recurring events, missing fields) through five helper functions.
  2. Security Caught a hardcoded API key in the source mid-development. Regenerated it on Humanitix and moved credentials into Wix Secrets Manager — key never touches the browser, never appears in any file.
  3. Performance 10-second TTL in-memory cache plus promise deduplication, so concurrent page loads share one in-flight API request rather than hammering Humanitix.
  4. Defensive coding Every $w() DOM call wrapped in a safeSet() guard, so missing elements never throw and stop the page render.
  5. Debugging Found a silent filter where e.published === true excluded every event (the field doesn't exist in the public API, so the comparison was always false). Removed it, added stage-by-stage console tracing.
What it taught me

Silent failures are harder to debug than thrown errors. The published filter never threw — it just quietly returned an empty array. Stage tracing through the data pipeline turned a day's hunt into a five-minute fix. Also: an API key hardcoded "just for testing" is a compromised key. Same time, no risk — use the secrets manager from line one.

DevOps Pipeline & Cloud Deployment Automation

2024 RMIT — Systems Deployment & Operations Distinction

Two assignments, one continuous theme — take a working application from "deploys manually if you remember the steps" to "anyone on the team can ship it with one PR merge."

The problem

Manual deployments are slow, error-prone, and impossible to audit. The unit's two assignments wanted a working CI/CD pipeline and a fully Infrastructure-as-Code deployment story for two different Node.js apps — one backed by MongoDB, one by PostgreSQL.

The approach

Treat the pipeline like a product in its own right: branch protections, automated tests on every pull request, and a deployment path defined entirely in version-controlled code, so it's reproducible from a clean machine.

What I did, specifically

  1. CI pipeline Built a GitHub Actions workflow for a Node.js / Express / MongoDB app under a GitHub Flow branching strategy. Every PR triggers install, lint, and test jobs before merge to main is allowed.
  2. Provisioning Wrote Terraform configs to spin up AWS EC2 instances and the surrounding network resources — versioned, reviewed via PR, destroyable with one command.
  3. Configuration Ansible playbooks handle the post-provision side: installing Docker, pulling images, wiring environment variables, starting the Node.js + PostgreSQL service.
  4. Containers Dockerised the apps so the developer environment, the CI runner, and the AWS host all see the same runtime.
What it taught me

Infrastructure-as-Code is the difference between "it deploys" and "anyone can deploy it." The first time you destroy your environment and rebuild it from a clean clone in fifteen minutes is genuinely satisfying. Also: separation of concerns matters here — Terraform for what exists, Ansible for what's installed on it, Docker for what runs inside it. Mixing those responsibilities gets messy quickly.

AWS Cloud Infrastructure Build

2024 RMIT — Cloud Foundations High Distinction

A full production-shaped AWS environment, built by hand in the console — networking, Linux servers, storage, databases, and an auto-scaling web tier. Not a tutorial follow-along: every resource was provisioned, connected, secured, and verified end to end.

The problem

Take a small business website running on a single server and re-platform it as a real cloud deployment: isolated networking across multiple availability zones, a private link to an on-premises network, encrypted traffic, durable storage, and a web tier that survives load and instance failure — all reproducible from the console.

The approach

Build it the way an operations team actually runs a data centre: network first, then compute, then storage and data, then resilience. Lock each layer down with least-privilege security groups and IAM, and prove every piece works before moving to the next.

What I did, specifically

  1. Network foundation Built a VPC spanning three availability zones — three public and three private subnets, separate public/private route tables, an internet gateway and a NAT gateway, and security groups scoped to HTTP, HTTPS and SSH only.
  2. Hybrid connectivity Linked the private subnets to a simulated on-premises network (192.168.10.0/24) with a site-to-site VPN — customer gateway, virtual private gateway, and the routing to make the two networks talk.
  3. Compute & Linux Launched Linux EC2 instances via user-data (Apache + MariaDB + PHP), assigned a persistent Elastic IP, connected over SSH, and verified the web and PHP stack from the command line.
  4. TLS & hardening Generated a private key and CSR with openssl, issued a self-signed certificate, enabled mod_ssl, configured an Apache SSL virtual host, and confirmed HTTPS served correctly.
  5. Storage operations Created, attached, formatted and mounted EBS volumes; took a snapshot, then restored and resized it onto a new volume; made mounts survive reboot via fstab; and shared an EFS file system across instances.
  6. Serverless, S3 & IAM Hosted a static site on S3, wired a Lambda trigger that logs new-object types to CloudWatch, and gave EC2 private access to a second bucket through a VPC endpoint and IAM — no public exposure.
  7. Databases Stood up a Multi-AZ RDS MySQL instance and a DynamoDB table, querying the latter with PartiQL.
  8. High availability Fronted the app with an Elastic Load Balancer and a target group, then put it behind an Auto Scaling Group with launch templates and dynamic scaling policies — load-tested it to watch scale-out and scale-in events fire, with SNS notifications on each.
What it taught me

Most of what people call "cloud" is really networking and disciplined access control. The compute is the easy part — the hours go into subnets, routing, security groups and IAM, and that's exactly where a misconfiguration quietly breaks everything or quietly leaves a door open. Watching an Auto Scaling Group add and drop instances on its own under load was the moment it clicked: resilient infrastructure isn't a single big server, it's a system designed to lose pieces and keep running.

Grad Job Tracker

2026 Personal project Try it live

A tool I built for my own graduate job hunt — because a spreadsheet wasn't cutting it. A focused application tracker you can actually use right now, in your browser.

The problem

Tracking dozens of applications across stages — applied, screening, interview — in a spreadsheet is clumsy and easy to let slip. I wanted something purpose-built that I'd actually open every day during the hunt.

The approach

Build it deliberately without a framework — clean state management, DOM rendering, and client-side persistence handled directly. Zero dependencies so it loads instantly and nothing can break.

What it does

  1. Pipeline board Six-stage kanban — Wishlist → Applied → Screening → Interview → Offer / Rejected — with one-click stage moves.
  2. Live stats Total tracked, active applications, and a computed interview-rate metric that updates as the data changes.
  3. Persistence Everything saves to localStorage — survives refreshes, no account, no backend, data stays on the user's device.
  4. Search & filter Live text search across company / role / location, plus stage filtering, with event-delegated rendering.
Why no framework

Reaching for React isn't always the answer. Built directly, this loads instantly, has zero dependencies, and nothing in the supply chain can break it. Knowing when not to add a framework is its own skill.
